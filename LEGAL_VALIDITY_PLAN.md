## Часть 2. Правовая база (Казахстан)

### Ключевые нормативные акты

1. **Трудовой кодекс РК от 23.11.2015 № 414-V**
   - Ст. 32: Трудовой договор может быть заключён в электронной форме
   - Ст. 33: Электронный трудовой договор подписывается ЭЦП **обеих сторон**
   - Ст. 35: Договор вступает в силу **с даты подписания обеими сторонами**
   - Ст. 11: Акты работодателя оформляются в электронной форме, удостоверенной ЭЦП

2. **ЗРК «Об электронном документе и электронной цифровой подписи» от 07.01.2003 № 370-II**
   - Ст. 7: Электронный документ с ЭЦП **равнозначен** документу на бумажном носителе
   - Ст. 11: Документ с ЭЦП presume неизменным с момента подписания

3. **ЕСУТД (enbek.kz)**
   - Обязательная регистрация трудовых договоров через Единую систему учёта
   - Работодатель обязан зарегистрировать договор

4. **ЗРК «О персональных данных и их защите»**
   - Согласие на обработку ПДн может быть получено в электронной форме

### Вывод

По закону РК **трудовой договор должен быть подписан ЭЦП обеих сторон** — и работодателя, и работника. Без подписи работодателя документ **юридически неполноценен**.

## Часть 3. Ответы на ключевые вопросы

### Вопрос 1: Может ли юрист подготовить документ как публичную оферту?

**Короткий ответ: Нет, для трудового договора это не подходит.**

**Развёрнутый ответ:**

Публичная оферта (ст. 395 ГК РК) — это предложение заключить договор **неопределённому кругу лиц**. Трудовой договор — это **индивидуальный** договор с конкретным лицом, с конкретными условиями (должность, оклад, место работы).

**Что МОЖНО сделать через оферту:**
- ✅ **Заявление на приём** — это по сути оферта работника работодателю. Работник пишет заявление → работодатель акцептует (издаёт приказ). Это корректная схема.
- ✅ **Согласие на обработку персональных данных** — может быть оформлено как акцепт оферты.
- ✅ **Правила внутреннего трудового распорядка (ПВТР)** — могут быть утверждены приказом и доведены под подпись (или через ЭЦП).

**Что НЕЛЬЗЯ сделать через оферту:**
- ❌ **Трудовой договор** — требует двустороннего подписания (ст. 33 ТК РК)
- ❌ **Приказы** — это односторонние акты работодателя, требуют ЭЦП директора
- ❌ **Дополнительные соглашения к договору** — требуют двустороннего подписания

### Вопрос 2: Как будет правильно?

**Правильная схема для каждого типа документа:**

#### Трудовой договор
```
HR генерирует → Работодатель подписывает ЭЦП (NCALayer) →
Работник подписывает ЭЦП (eGov Mobile) →
Договор вступает в силу → Регистрация в ЕСУТД (enbek.kz)
```

#### Приказ о приёме
```
HR генерирует → Работодатель подписывает ЭЦП (NCALayer) →
Работник ознакомлен (может подписать ЭЦП подтверждение ознакомления)
```

#### Заявление на приём
```
Работник заполняет → Работник подписывает ЭЦП (eGov Mobile) →
Работодатель получает (акцепт — издание приказа)
```

## Часть 4. Технический план реализации

### Фаза 1: Подписание работодателем (2-3 дня)

**Цель:** Работодатель (директор/ИП) подписывает документы через NCALayer

#### 1.1. База данных

```sql
-- Добавить колонку для подписи работодателя
ALTER TABLE documents
ADD COLUMN IF NOT EXISTS signature_cms_employer TEXT,
ADD COLUMN IF NOT EXISTS employer_signed_at TIMESTAMP WITH TIME ZONE,
ADD COLUMN IF NOT EXISTS employer_cert_info JSONB;

-- Для документов, требующих двух подписей
ALTER TABLE documents
ADD COLUMN IF NOT EXISTS requires_employer_signature BOOLEAN DEFAULT FALSE;

-- Типы документов, требующие подписи работодателя
UPDATE documents SET requires_employer_signature = TRUE
WHERE type IN ('contract', 'order_hiring', 'vacation_order', 'termination_order');
```

#### 1.2. Frontend — режим подписания для HR/директора

- В `DocumentsList.tsx` добавить кнопку **«Подписать как работодатель»**
- При нажатии → NCALayer (только этот метод, т.к. у директора ключ на ПК)
- После подписания → `signature_cms_employer` сохраняется в БД
- Визуальный статус:
  - `draft` — никто не подписал
  - `employer_signed` — работодатель подписал, ждёт работника
  - `employee_signed` — работник подписал, ждёт работодателя
  - `signed` — обе подписи ✅

#### 1.3. Backend — эндпоинт подписания работодателем

```
POST /documents/:id/sign-employer
Body: { signature: "CMS_BASE64", certInfo: {...} }
```

#### 1.4. SigexSignModal — поддержка двух подписей

- Добавить проп `signingRole: 'employer' | 'employee'`
- Для `employer` — только NCALayer (нет QR, т.к. у директора ключ на ПК)
- Для `employee` — текущий флоу (eGov QR + NCALayer)

### ТЕХНИЧЕСКОЕ ЗАДАНИЕ
### на разработку/актуализацию шаблонов кадровых документов
### для системы электронного документооборота PickPoint OS

#### 1. Контекст

PickPoint OS — корпоративная система управления персоналом для сети ПВЗ (пунктов выдачи заказов). Система генерирует кадровые документы в электронном виде и обеспечивает их подписание ЭЦП (электронной цифровой подписью) через интеграцию с Sigex.kz (национальная платформа ЭЦП РК).

**Документы подписываются квалифицированной ЭЦП** от НУЦ РК (Национальный удостоверяющий центр), что обеспечивает полную юридическую значимость согласно ЗРК «Об электронном документе и электронной цифровой подписи».

#### 2. Что нужно сделать

Разработать/актуализировать **шаблоны следующих документов**:

| № | Документ | Тип | Стороны |
|---|----------|-----|---------|
| 1 | Трудовой договор | Двусторонний договор | Работодатель (ИП) + Работник |
| 2 | Приказ о приёме на работу | Односторонний акт работодателя | Работодатель (директор), работник — ознакомлен |
| 3 | Заявление на приём на работу | Оферта работника | Работник → Работодатель |
| 4 | Заявление на предоставление отпуска | Заявление | Работник → Работодатель |
| 5 | Приказ о предоставлении отпуска | Односторонний акт | Работодатель, работник — ознакомлен |
| 6 | Приказ об увольнении | Односторонний акт | Работодатель |
| 7 | Справка с места работы | Информационный документ | Работодатель (выдаёт) |

#### 3. Требования к шаблонам

##### 3.1. Общие требования

1. **Все документы** должны соответствовать **действующему законодательству РК** (Трудовой кодекс, ЗРК об ЭЦП, ЗРК о персональных данных)
2. **Двуязычность**: каждый документ должен быть представлен на **русском и казахском языках** (параллельные колонки или последовательные блоки)
3. **Электронная форма**: документы предназначены **исключительно для электронного документооборота** с подписанием ЭЦП
4. **Переменные**: в шаблонах должны быть чётко выделены места для подстановки данных (в системе они обозначаются как `{{variable_name}}`)

##### 3.2. Трудовой договор (приоритет №1)

**Обязательные элементы:**

1. **Преамбула:**
   - Полное наименование работодателя (ИП/ТОО)
   - ФИО директора/представителя
   - БИН/ИИН работодателя
   - ФИО работника, ИИН, данные удостоверения личности

2. **Предмет договора:**
   - Должность (с указанием на казахском и русском)
   - Место работы (конкретный адрес ПВЗ)
   - Ссылка на ст. 32, 33, 35 ТК РК

3. **Срок действия:**
   - Дата начала работы (параметризируемая)
   - Дата окончания (для срочного) ИЛИ «бессрочный»
   - Испытательный срок (параметризируемый, до 3 месяцев)

4. **Права и обязанности сторон:**
   - Актуальные ссылки на ТК РК (редакция 2025-2026)
   - Условия труда и охрана труда
   - Режим работы и отдыха
   - Оплата труда (базовая ставка + премии)

5. **Ответственность сторон:**
   - Материальная ответственность
   - Дисциплинарная ответственность

6. **Заключительные положения:**
   - **Обязательно:** «Настоящий договор заключён в электронной форме и подписан электронными цифровыми подписями сторон в соответствии со ст. 32, 33 ТК РК и ЗРК «Об электронном документе и электронной цифровой подписи»»
   - Порядок изменения и расторжения
   - Порядок разрешения споров

7. **Блок подписей:**
   - Не «_________________ (подпись)», а:
   - «Подписано ЭЦП Работодателя: [данные из сертификата]»
   - «Подписано ЭЦП Работника: [данные из сертификата]»
   - Дата подписания (автоматически из системы)

##### 3.3. Приказ о приёме на работу

1. Шапка: наименование работодателя, дата, номер приказа
2. Текст: «ПРИКАЗЫВАЮ: принять [ФИО] на должность [должность] с [дата]»
3. Основание: «Трудовой договор № [номер] от [дата]»
4. Подпись директора: «Подписано ЭЦП»
5. Ознакомление работника: «Ознакомлен(-а) ЭЦП» или «Ознакомлен(-а) под подпись»

##### 3.4. Заявление на приём

1. Шапка: «Директору [наименование] от [ФИО работника]»
2. Текст: «Прошу принять меня на работу с [дата] на должность [должность]»
3. Подпись: «Подписано ЭЦП» (не бумажная подпись!)
4. Дата: автоматически из системы

##### 3.5. Остальные документы

Аналогичные требования:
- Убрать все бумажные подписи «_________________»
- Заменить на «Подписано ЭЦП» / «Ознакомлен(-а) ЭЦП»
- Актуальные ссылки на законодательство
- Двуязычность (RU/KZ)

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Юрист не знаком с ЭЦП/КЭДО | Средняя | Высокое | Предоставить этот документ + примеры |
| Задержка с API enbek.kz | Высокая | Среднее | Начать запрос доступа заранее |
| Шаблоны не влезут в HTML | Низкая | Низкое | Использовать Puppeteer для сложных layout |

| Изменения в ТК РК | Низкая | Высокое | Мониторить, шаблоны в БД = быстрое обновление |

## Приложение: Что сказать юристу при передаче ТЗ

> «Мы — IT-компания, разрабатываем систему электронного кадрового документооборота для сети ПВЗ в Казахстане.
>
> Система генерирует документы (трудовые договоры, приказы, заявления) и позволяет подписывать их **квалифицированной ЭЦП** через интеграцию с Sigex.kz (национальный провайдер ЭЦП). ЭЦП выдаётся НУЦ РК и имеет полную юридическую силу.
>
> **Проблема:** сейчас документы подписывает только работник. По закону (ст. 33 ТК РК) трудовой договор должен быть подписан **обеими сторонами**. Нам нужно:
>
> 1. **Актуализировать шаблоны** документов — убрать бумажные подписи, заменить на ЭЦП-блоки, актуализировать ссылки на законодательство
> 2. **Двуязычность** — каждый документ на RU и KZ
> 3. **Корректная юридическая структура** — особенно для трудового договора
>
> **НЕ нужна публичная оферта** для трудового договора — это индивидуальный договор с конкретным лицом. Оферта подходит только для заявления работника.
>
> Техническое задание с перечнем документов и требованиями — в приложении.»
