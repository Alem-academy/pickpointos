# Миграция: Привязка юрлиц к ПВЗ и сотрудникам

**Дата:** 2026-03-26  
**Цель:** Автоматическое определение юрлица (employer) на основе ПВЗ, к которому привязан сотрудник.

## Проблема

Раньше юрлицо для генерации документов выбиралось вручную или по умолчанию (ИП «Жасмин»).  
Фактически каждый ПВЗ принадлежит определенному юрлицу:
- **ИП Жасмин** - 58 ПВЗ (Алматы, Актобе, Караганда и др.)
- **ИП Ориентал** - 13 ПВЗ (Астана, Атырау, Рудный, Сатпаев, Актау)
- **ТОО PVZ.kz** - 1 ПВЗ (Астана)

## Решение

1. Добавлено поле `employer_id` в таблицу `pvz_points`
2. Добавлено поле `employer_id` в таблицу `employees` (кэшируется из ПВЗ)
3. Создан триггер для авто-обновления `employee.employer_id` при смене `main_pvz_id`
4. Генерация документов автоматически использует юрлицо из ПВЗ сотрудника

### Шаг 3: Проверка

```sql
-- Проверить распределение ПВЗ по юрлицам
SELECT 
    e.name_short as employer,
    COUNT(p.id) as pvz_count
FROM pvz_points p
JOIN employers e ON p.employer_id = e.id
GROUP BY e.name_short
ORDER BY pvz_count DESC;

-- Проверить распределение сотрудников по юрлицам
SELECT 
    e.name_short as employer,
    COUNT(emp.id) as employee_count
FROM employees emp
JOIN employers e ON emp.employer_id = e.id
GROUP BY e.name_short
ORDER BY employee_count DESC;
```

## Изменения в коде

### Backend (`parser/src/routes/`)

### Frontend (`src/`)

#### `api/hr.ts`
- Метод `getEmployers()` уже существует и готов к использованию

## Как это работает

### При приеме сотрудника на работу

1. HR выбирает ПВЗ из списка (например, "Алматы, ул. Жарокова, 5129")
2. Система автоматически определяет юрлицо (ИП «Жасмин»)
3. При генерации документов используется это юрлицо

### При переводе сотрудника на другой ПВЗ

1. HR меняет `main_pvz_id` сотрудника
2. Триггер автоматически обновляет `employer_id`
3. Новые документы генерируются с новым юрлицом

### Генерация документов

```javascript
// Frontend вызывает API
await hrApi.generateDocument(employeeId, 'contract');

// Backend автоматически определяет юрлицо:
// 1. Получает employee.main_pvz_id
// 2. Находит pvz_points.employer_id
// 3. Получает employers.реквизиты
// 4. Генерирует документ с правильным юрлицом
```

При необходимости отката:

```sql

DROP TRIGGER IF EXISTS trg_update_employee_employer ON employees;

-- Удалить функцию
DROP FUNCTION IF EXISTS update_employee_employer();

-- Удалить поля (осторожно! потеря данных)
ALTER TABLE employees DROP COLUMN IF EXISTS employer_id;
ALTER TABLE pvz_points DROP COLUMN IF EXISTS employer_id;

-- Удалить таблицу employers (если нужна)
DROP TABLE IF EXISTS employers CASCADE;
```

## Тестирование

1. Создать нового сотрудника, привязать к ПВЗ (ИП Ориентал)
2. Сгенерировать трудовой договор
3. Проверить, что в шапке документа указано правильное юрлицо
4. Перевести сотрудника на другой ПВЗ (ИП Жасмин)
5. Сгенерировать новый документ
6. Убедиться, что юрлицо изменилось

## Миграция данных

После применения миграции все существующие сотрудники получат `employer_id` на основе своего `main_pvz_id`.

Сотрудники без указанного ПВЗ останутся без юрлица (NULL) - для них будет использоваться fallback (ИП «Жасмин»).
